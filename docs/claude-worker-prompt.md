# Claude 协作提示词

把下面整段发给 Claude。这个提示词是为当前 STM32F407ZGT6 项目定制的，重点是让 Claude 做适合它做的事，避免误改硬件相关和时序敏感代码。

```text
你现在接手一个 STM32F407ZGT6 嵌入式项目：“双手手语翻译手套”（v3.1）。

先不要急着改代码，先遵守下面的工作边界。

一、你的任务目标
1. 帮我做静态代码审查，优先找“能编过但真机可能出问题”的逻辑风险。
2. 帮我清理文档和注释，尤其是中文乱码、旧模块命名、过时注释。
3. 帮我整理硬件映射和软件实现之间的一致性，输出差异表。
4. 如果要改代码，只允许做小范围、可解释、低风险修复。

二、你绝对不能做的事
1. 不要擅自改 STM32 引脚分配。
2. 不要擅自改 CubeMX 生成的初始化逻辑，除非我明确授权。
3. 不要因为“看起来更合理”就把 TIM、ADC、USART、DMA、GPIO 重构掉。
4. 不要重写整个 `main.c`、`gesture.c`、`soft_i2c.c`。
5. 不要把项目改成 RTOS。
6. 不要引入新第三方库。
7. 不要删除现有防御性保护，例如超时、冻结、冷却、边界钳位。
8. 不要假设现在已经有真实硬件连线。当前仍处于硬件采购阶段。
9. 不要把“文档期望引脚”直接覆盖成“代码当前引脚”，也不要反过来直接改代码，先输出差异。

三、你必须先理解的项目事实
1. 当前项目没有实际硬件联机验证条件，重点是做“无硬件阶段固件收敛”。
2. 当前代码可以烧录，近期构建记录显示无报错，但这不代表运行逻辑一定正确。
3. 这个项目是多速率裸机调度，不是普通单循环程序。
4. `TIM3 1ms` 是全局调度时钟，不能随便动。
5. `gesture.c` 是手势识别核心，时序和状态机都比较敏感。
6. `soft_i2c.c`、`max30102.c`、`dfplayer.c`、`bluetooth.c` 都属于“容易编译通过、但容易埋运行时坑”的模块。

四、你应该优先做的事情
第一优先级：只读审查
1. 逐个检查这些文件是否存在“编译通过但运行危险”的问题：
   - `shuangshou/Core/Src/main.c`
   - `shuangshou/Core/Src/gesture.c`
   - `shuangshou/Core/Src/max30102.c`
   - `shuangshou/Core/Src/dfplayer.c`
   - `shuangshou/Core/Src/bluetooth.c`
   - `shuangshou/Core/Src/vibrator.c`
2. 输出发现的问题时，必须按下面格式：
   - 严重级别
   - 文件路径
   - 行为风险
   - 为什么这是问题
   - 建议修复方式

第二优先级：文档修正
1. 清理注释中的中文乱码。
2. 检查是否还保留过时命名，比如老的 MPU6050 说法、旧版本注释、和 JY61P 不一致的说明。
3. 生成“代码现状说明”，说明当前各模块实际在做什么，而不是理想设计要做什么。

第三优先级：硬件一致性核对
1. 对照 `.ioc`、`main.c`、各外设驱动，整理实际使用到的外设和引脚。
2. 输出“代码当前实现”和“设计文档描述”的差异表。
3. 对于差异，只允许标注，不允许未经批准直接修改。

五、你改代码时必须遵守的规则
1. 每次只改一类问题。
2. 改动必须小，不要顺手大改格式。
3. 先解释改动目的，再给出改动内容。
4. 如果一个问题涉及硬件未定、接线未定、文档矛盾，先不要改代码，先停下来汇报。
5. 如果某个修复会影响手势识别阈值、时序周期、中断行为、DMA 行为，必须单独说明风险。

六、你特别适合帮我做的任务
1. 抓注释和文档乱码并修正。
2. 找状态机遗漏分支。
3. 找字符串缓冲区、DMA 缓冲区、环形缓冲区边界问题。
4. 找初始化不完整、状态复位不完整的问题。
5. 帮我梳理蓝牙命令、手势词表、模式切换逻辑。
6. 帮我写“测试清单”和“联调步骤文档”。

七、你暂时不要做的任务
1. 不要改手势词库内容。
2. 不要新增复杂功能。
3. 不要改 Web 页面交互逻辑，除非我明确要求。
4. 不要改板级供电设计。
5. 不要自己发明新的硬件连接方案。

八、你的输出格式
先给我：
1. 你理解到的当前项目状态
2. 你准备处理的范围
3. 你发现的前 3 个最高优先级问题

如果没有得到我许可，不要直接大面积改代码。
如果要改，只提交最小补丁。
```

## 推荐你给 Claude 分配的具体活

最适合现在立刻交给 Claude 的有 3 类：

1. 注释和文档乱码清理
目标文件：
- `shuangshou/Core/Src/main.c`
- `shuangshou/Core/Src/gesture.c`
- `README.md`
- 根目录现有的几份 `.md`

2. 硬件映射一致性表
目标：
- 对比设计文档、`.ioc`、代码实际调用
- 输出“已一致 / 未一致 / 待硬件确认”三栏

3. 纯静态风险审查
重点看：
- DMA 生命周期
- 状态机是否有卡死路径
- 缓冲区越界
- 初始化和复位是否完整
- 中断和主循环共享变量是否安全

## 不建议现在让 Claude 碰的内容

- `.ioc` 直接改引脚
- 大改 `main.c` 调度结构
- 重构手势算法
- 改软 I2C 基础实现
- 把项目迁移到别的框架
