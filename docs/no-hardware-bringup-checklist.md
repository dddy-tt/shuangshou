# 无硬件阶段联调清单

当前阶段目标不是“验证传感器真值”，而是把固件整理到适合后续上板联调的状态。下面这份清单按优先级排列。

## 1. 当前已确认的代码链路

1. `TIM3` 作为 `sys_tick_ms` 的 1ms 调度时钟在主框架中存在。
2. 50ms 手势识别任务会调用 `Gesture_Evaluate()`。
3. 手势识别结果会分流到：
   - 翻译/康复模式：`DFPlayer_Play(...)`
   - 控制模式：`SoftUART_SendMQTT(...)`
4. 模式切换逻辑为双手全握 3 秒，通过 `Gesture_Freeze()` / `Gesture_Unfreeze()` 防止寄生触发。
5. 蓝牙校准命令和灵敏度命令现在都已经接入主循环消费。

## 2. 当前最值得做的无硬件检查

### 蓝牙命令链

检查目标：
- 蓝牙能否正确形成命令
- 主循环能否正确消费命令
- 消费后是否真的改到算法状态

应核对：
- `BT_CMD_CAL_MIN`
- `BT_CMD_CAL_MAX`
- `BT_CMD_SENS_1`
- `BT_CMD_SENS_2`
- `BT_CMD_SENS_3`

通过标准：
- 协议定义、接收解析、主循环消费、算法入口四段链路全部存在

### 手势模式链

检查目标：
- 3 种模式的切换逻辑完整
- 切换过程中不会误触发语音或控制

应核对：
- `MODE_TRANSLATE`
- `MODE_CONTROL`
- `MODE_REHAB`
- `Gesture_Freeze()`
- `Gesture_Unfreeze()`
- `DFPlayer_Stop()`

通过标准：
- 进入切换、切换完成、提前松手 3 个分支都能自洽

### 语音播报链

检查目标：
- DMA 发送缓冲生命周期安全
- 忙状态下不会引用失效内存
- 忙状态下新指令不会直接丢栈变量

应核对：
- `DFPlayer_Play()`
- `DFPlayer_Stop()`
- `DFPlayer_DMA_TxCplt()`
- `DFPlayer_IsBusy()`

通过标准：
- 发送帧为静态或持久化缓冲
- DMA 完成前数据不会失效

### MAX30102 处理链

检查目标：
- 批处理时心率间隔计算尽量接近真实时间
- 手指离开后历史状态能正确复位

应核对：
- FIFO 入环时是否带时间戳
- 峰值间隔是否基于样本时间
- `ResetHistory()` 是否把峰值状态清干净

通过标准：
- 重新佩戴后不会沿用旧 HR/SpO2 历史

## 3. 需要保守处理的硬件差异

下面这些点先记录，不建议现在直接改代码：

1. 振动电机 PWM
当前代码/工程更像是 `TIM4 PD12/PD13` 两路方案，但设计资料里写过 `PE2..PE5` 四路方案。

2. ESP-01S 软串口
代码里已有 `PE0` 软串口发送逻辑，但 `.ioc` 不一定完整体现。

3. 蓝牙模块型号
项目目标是 `JDY-33`，不是 `JDY-31`。这属于采购和联机条件，不属于当前固件重构范围。

## 4. 上板前建议准备的测试顺序

等硬件到位后，建议按这个顺序联调，不要一上来全开。

1. 先验证 `TIM3` 调度是否运行。
2. 再验证蓝牙收命令。
3. 再验证 DFPlayer 单独播报。
4. 再验证 ADC 柔性传感器采样。
5. 再验证单个 JY61P。
6. 再验证双 JY61P。
7. 再验证 MAX30102。
8. 最后再联通 ESP-01S MQTT。

原因：
- 这样能把“时钟问题、通信问题、采样问题、算法问题”分层拆开。

## 5. 你现在就可以人工检查的事项

即使没有硬件，也可以人工确认这些：

1. 所有状态机是否都有退出路径。
2. 所有 `static` 状态是否在初始化或复位函数中归零。
3. 所有 DMA 使用是否只引用长期有效缓冲区。
4. 所有字符串接收缓冲区是否有边界保护和结尾补零。
5. 所有 `sys_tick_ms` 替换是否统一，避免和 `HAL_GetTick()` 混用产生节拍漂移。
6. 所有模式切换和报警逻辑是否会互相打断。
